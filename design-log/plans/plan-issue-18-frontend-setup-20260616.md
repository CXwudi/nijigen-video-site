# Issue 18 Frontend Setup Implementation Plan

> **For agentic workers:** Use the harness's preferred task-tracking and delegation tools when available. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Set up the first real frontend application under `frontend/`, replacing the temporary frontend Docker placeholder with a Docker-first TanStack Start workflow that also supports host-run development, production-like Compose, and CI checks.

**Source of Truth:** Approved design spec `design-log/spec/spec-issue-18-frontend-setup-20260614.md`, GitHub issue `#18`, and the existing component-owned Docker module guidance.

**Scope:** Includes the `frontend/` pnpm workspace, initial `frontend/web/` app, selected frontend stack and tooling, Snow Miku daisyUI theme integration, frontend Dockerfile stages, local and production-like Compose wiring, frontend `mise` tasks, frontend CI, and workflow documentation. Excludes unrelated backend implementation changes, multiple frontend apps beyond future-proof naming, and production hardening beyond the existing production-like Compose model.

**Approach:** Implement from the inside out: first create a working host-run frontend workspace, then containerize it for local development, then connect production-like Compose and CI around the same boundaries. Keep shared Compose bases minimal and place concrete stack resources in the caller Compose file, matching the repository's existing Docker model.

**Verification:** Validate host-run frontend commands, local Docker Compose config and startup paths, production-like Compose config and web image build, frontend CI commands, and documentation links.

## File and Boundary Map

- `frontend/package.json`, `frontend/pnpm-workspace.yaml`, `frontend/pnpm-lock.yaml`, `frontend/mise.toml`: new frontend-owned pnpm workspace root and mise task delegation.
- `frontend/web/`: new first frontend application using TanStack Start, React, Tailwind CSS, daisyUI, TanStack Query, Zustand, Vite, Vitest, Oxlint, Oxfmt, TypeScript, and React.
- `frontend/docker/Dockerfile`: new frontend-owned Dockerfile with `web-node-base`, `web-pnpm-base`, `web-build`, and `web-runtime` stages.
- `frontend/docker/compose.local.yml`: replace the temporary `nginx` web service with the real `web` service, keep explicit backend dependency services, and declare frontend dependency volumes.
- `frontend/docker/mise.toml`: update frontend Docker tasks for `up`, `up-backend`, `down`, `config`, `config-check`, `clean-dependency-volumes`, and `run`.
- `frontend/docker/.env.example`: remove dummy web image settings and add frontend web Docker settings needed by the local stack.
- `infra/compose/common-services.yml`: add `web-compose-base` for common web runtime contract and API dependency wiring.
- `infra/compose/common-dev-services.yml`: add `web-pnpm-base` for source-mounted pnpm development behavior.
- `infra/compose/compose.prod.yml` and `infra/compose/prod.env.example`: add production-like `web` service and web production env values.
- `.github/workflows/`: add frontend CI workflow, likely `frontend-check.yml`, using Docker/Compose as the environment boundary and Docker layer caching for frontend builds.
- `docs/`, `frontend/README.md`, `frontend/docs/`: update documentation after the runnable commands are verified.
- `mise.toml`: add `frontend` as a monorepo config root when `frontend/mise.toml` is introduced; keep tool declarations aligned with the current root `node = "lts"` and `pnpm = "latest"` setup.

## Risks and Decision Points

- TanStack Start's generated output path, runtime command, and default port must be discovered during implementation and then propagated to Dockerfile and Compose.
- The exact current latest GA dependency versions must be checked during implementation, not guessed from memory.
- pnpm named volumes can hide newer image-layer dependency contents after lockfile changes, so the cleanup task and `--prefer-offline` startup behavior matter.
- Docker pnpm hardlink behavior may require `package-import-method=copy` inside Docker.
- The initial web-to-api environment contract may be tiny; avoid inventing public env vars before the app has real API calls.
- CI should favor Docker layer caching and avoid trying to make Compose named volumes persistent across runs.
